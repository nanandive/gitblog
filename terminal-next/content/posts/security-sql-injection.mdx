회사 코드 리뷰에서 시니어 개발자가 "이 부분 SQL Injection 취약합니다"라는 코멘트를 남겼다.

나는 그 의미를 알면서도 왜 취약한지, 어떻게 막는지는 사실 제대로 몰랐다. 강의를 보면서 실제로 코드로 확인해봤다.

## SQL Injection이 발생하는 이유

핵심은 간단하다. **쿼리 문자열과 사용자 입력 데이터가 구분되지 않는다는 것.**

```python
# 취약한 코드 (Python)
name = input("이름 입력: ")
age = int(input("나이 입력: "))

sql = f"INSERT INTO users (name, age) VALUES ('{name}', {age})"
cursor.execute(sql)
```

이름에 `admin', 20); DROP TABLE users; --`를 입력하면?

```sql
INSERT INTO users (name, age) VALUES ('admin', 20); DROP TABLE users; --', 입력한나이)
```

- `'admin'` → 이름 입력 완료
- `)` → INSERT 괄호 닫힘
- `;` → SQL 구문 종료
- `DROP TABLE users;` → 테이블 삭제 실행
- `--` → 이후 SQL 주석 처리

데이터가 명령어로 해석됐다.

## 왜 이름 필드에서 탈출할 수 있나

SQL에서 문자열은 따옴표(')로 감싼다. `'admin'`이면 admin이 문자열이다.

사용자 입력에 따옴표를 넣으면 SQL 파서 입장에서는 문자열 종료로 인식한다. 그 이후에 나오는 내용은 더 이상 문자열이 아니고 SQL 명령이 된다.

Python 같은 고수준 언어는 변수와 값이 명확히 구분되어 있다. `f"INSERT INTO ... VALUES ('{name}')"` 이 코드에서 Python은 `name` 변수를 문자열로 다룬다. 하지만 이걸 SQL로 전달하면 SQL 파서는 그냥 하나의 거대한 문자열로 받아서 해석한다.

## 해결책 1: 입력값 필터링

특수문자를 필터링하거나 이스케이프한다.

```python
# 따옴표를 이스케이프 (단순 방어)
name = name.replace("'", "''")
```

문제: 완벽한 필터링이 어렵다. 새로운 우회 방법이 계속 나온다.

## 해결책 2: Prepared Statement (권장)

데이터가 들어갈 자리(placeholder)와 실제 데이터를 분리한다.

```python
# Prepared Statement
sql = "INSERT INTO users (name, age) VALUES (?, ?)"
cursor.execute(sql, (name, age))
```

SQL 파서가 먼저 쿼리 구조를 파싱한다. `?` 자리가 어디인지 확정한다. 이후에 실제 데이터를 바인딩한다.

데이터는 절대로 SQL 명령어로 해석되지 않는다. 데이터와 명령어가 물리적으로 분리된다.

아무리 악의적인 입력이 들어와도 그냥 문자열 데이터로만 취급된다.

## Spring에서는 어떻게 되나

Spring JPA나 MyBatis를 쓰면 기본으로 Prepared Statement가 사용된다.

```java
// JPA - user값이 어떻든 안전
userRepository.findByEmail(email);

// MyBatis - #{}는 Prepared Statement
SELECT * FROM users WHERE email = #{email}
// 즉, SELECT * FROM users WHERE email = ?  로 동작

// 주의! ${}는 문자열 바인딩이라 취약
// SELECT * FROM users WHERE ${columnName} = #{value}
// columnName에 SQL을 주입할 수 있음
```

JPA JPQL에서도 직접 문자열을 연결하면 위험하다:

```java
// 취약
em.createQuery("SELECT u FROM User u WHERE u.email = '" + email + "'")

// 안전 (파라미터 바인딩)
em.createQuery("SELECT u FROM User u WHERE u.email = :email")
  .setParameter("email", email)
```

## OWASP Top 10에 항상 등장하는 이유

SQL Injection은 수십 년 된 공격인데 왜 아직도 발생하는가?

개발자가 Prepared Statement를 쓰지 않거나 레거시 코드에 남아있거나, 잠깐의 방심으로 직접 문자열 연결을 한다.

그리고 SQL만의 문제가 아니다. LDAP, XPath, OS 명령어 등 명령어와 데이터를 혼용하는 모든 곳에서 Injection 취약점이 생긴다.

원칙은 같다: **명령어와 데이터를 항상 분리하라.**

## 회고

"JPA 쓰면 SQL Injection 걱정 없다"는 말을 들었는데 반만 맞는 말이었다.

JPA의 기본 메서드(`findById`, `findByEmail` 등)는 안전하다. 하지만 JPQL 문자열 연결, Native Query 사용, MyBatis의 `${}` 등 직접 SQL을 다루는 순간부터 위험하다.

Injection이 왜 발생하는지 원리를 이해해야 내가 작성한 코드가 안전한지 판단할 수 있다.
