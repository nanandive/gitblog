회사에서 CICD 파이프라인을 구축하다가 서비스 간 연결이 간헐적으로 끊기는 문제를 만났다.

로그에는 `Connection reset by peer`, `Broken pipe` 같은 에러들. 네트워크를 모르고는 이 에러들이 왜 나는지 이해할 수가 없었다. 그래서 처음부터 파고들었다.

## 에펠탑을 택배로 보내려면

TCP/IP 송수신 구조를 이해하는 핵심 비유가 있다.

에펠탑을 파리에서 부산으로 택배로 보내려면?

1. **분해**: 운반 가능한 크기로 쪼갠다 (세그멘테이션)
2. **포장**: 각 조각을 박스에 넣는다 (캡슐화)
3. **운송**: 택배로 보낸다 (네트워크 전송)
4. **조립**: 받은 쪽에서 다시 조립한다 (디캡슐화)

실제 데이터 전송도 똑같다.

## 전체 흐름

```
[서버 프로세스]
    |
    | 파일 읽기 (e.g., 1.4MB 이미지)
    ↓
[프로세스 버퍼] ← 개발자가 크기 결정
    |
    | Buffered I/O
    ↓
[소켓 I/O 버퍼] ← OS가 관리
    |
    | send() 호출 → 유저 모드에서 커널로
    ↓
[TCP 스택 (L4)]
    |
    | 세그멘테이션 (MTU 기준으로 쪼개기)
    | 각 세그먼트에 순서 번호 부여 (1, 2, 3...)
    ↓
[IP 계층 (L3)]
    |
    | 세그먼트를 IP 패킷으로 포장
    | 출발지/목적지 IP 헤더 추가
    ↓
[L2 (Ethernet Frame)]
    |
    | 패킷을 프레임으로 포장
    | MAC 주소 추가
    ↓
[네트워크 전송]
    |
    | 라우터를 거칠 때마다 프레임은 바뀜
    | 패킷(IP)은 목적지까지 유지됨
    ↓
[수신 측] — 역순으로 디캡슐화
    |
    | 프레임 → 패킷 → 세그먼트 → 데이터
    ↓
[수신 측 소켓 I/O 버퍼]
    |
    | TCP가 버퍼 채움
    | 프로세스가 receive()로 꺼냄
    ↓
[수신 프로세스 버퍼]
```

## 버퍼가 핵심이다

왜 버퍼가 필요한가? **속도 차이 때문이다.**

- 네트워크는 계속 데이터를 받아서 소켓 버퍼를 채운다
- 프로세스는 소켓 버퍼에서 데이터를 꺼내서 처리한다

이 두 작업이 동시에 일어난다. 버퍼가 중간에서 속도 차이를 흡수해준다.

소켓 버퍼가 가득 차면? 송신 측에 "지금은 더 보내지 마"라는 신호가 간다. 이게 **흐름 제어(Flow Control)** 다.

## ACK와 재전송

TCP는 연결 지향(Connection-Oriented)이다. 받은 쪽은 잘 받았다고 피드백을 준다.

```
송신: 세그먼트 1, 2 전송
수신: 잘 받았어, 3번부터 줘 (ACK = 3)

송신: 세그먼트 3, 4 전송
수신: ... (응답 없음)

송신: 타임아웃 → 재전송 (세그먼트 3, 4 다시)
```

`Connection reset by peer`는 상대방이 RST 패킷을 보낸 것. 연결을 강제로 끊는 신호다.

`Broken pipe`는 연결이 끊어진 소켓에 데이터를 쓰려 했을 때 발생한다.

이 에러들의 원인: 타임아웃, 방화벽 설정, 半-열린 연결(Half-open Connection) 등.

## 프레임과 패킷의 차이

헷갈렸던 부분이다.

**패킷 (IP Packet)**: 출발지에서 목적지까지 L3 레벨의 데이터 단위. IP 주소가 담겨있다. 경로 중간에 바뀌지 않는다.

**프레임 (Ethernet Frame)**: L2 레벨. MAC 주소가 담겨있다. 라우터를 거칠 때마다 프레임이 새로 만들어진다.

택배 비유: 박스 속 물건(패킷)은 목적지까지 동일하지만, 운반하는 트럭(프레임)은 물류센터마다 바뀐다.

## CI/CD 파이프라인 문제 해결

알고 보니 배포 중에 롤링 업데이트로 인스턴스가 교체되면서 기존 TCP 연결이 강제로 끊겼던 거였다.

해결책:
- Health check 엔드포인트 구성
- Graceful Shutdown 설정 (기존 연결 처리 완료 후 종료)
- Connection Draining 타임아웃 설정

```yaml
# Spring Boot Graceful Shutdown
server:
  shutdown: graceful

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 30초 동안 기존 요청 처리 후 종료
```

TCP 구조를 이해하고 나서야 왜 이런 설정이 필요한지 이해됐다.
