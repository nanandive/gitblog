회사에서 대규모 ETL 시스템을 개발하다가 특정 배치 작업에서 쿼리 성능이 갑자기 뚝 떨어지는 현상을 만났다.

처음엔 인덱스 문제인가 싶었는데, 알고 보니 버퍼풀이 풀 테이블 스캔에 의해 오염되고 있었던 거였다. 그때부터 MySQL 내부 구조를 제대로 파고들기 시작했다.

## MySQL 메모리 구조: 전역 vs 세션

MySQL의 메모리는 크게 **전역 메모리**와 **세션 메모리**로 나뉜다.

**전역 메모리**는 모든 연결이 공유하는 공간이다.
- 버퍼 풀 (Buffer Pool)
- 로그 버퍼
- 테이블 캐시

**세션 메모리**는 연결마다 할당되는 메모리다.
- 정렬 버퍼: ORDER BY, GROUP BY 작업 시 사용
- 조인 버퍼: 인덱스 없는 조인 작업
- 읽기 버퍼: 테이블 풀 스캔 시 사용

세션 메모리는 연결 수가 많을수록 총 메모리 사용량이 폭증한다. 커넥션 풀 사이즈를 막 늘리다가 OOM을 맞는 이유가 여기 있다.

## 버퍼풀 — MySQL의 심장

솔직히 MySQL에서 제일 중요한 메모리 영역 딱 하나만 꼽으라면 **버퍼풀**이다.

자주 사용하는 데이터를 메모리에 캐싱해서 디스크 I/O를 줄이는 게 핵심이다. 이게 잘 작동하면 쿼리 응답 속도가 극적으로 빨라진다.

기본적으로 **LRU 알고리즘**을 사용한다. 가장 최근에 사용되지 않은 데이터를 먼저 제거하는 방식.

근데 순수 LRU는 문제가 있다.

```sql
-- 이런 풀 스캔 쿼리 한 방이면...
SELECT * FROM orders; -- 수천만 건

-- 자주 쓰던 캐시 데이터가 다 밀려나버린다
```

전체 테이블 스캔이 발생하면 그 쿼리에서 읽은 대용량 데이터가 버퍼풀을 가득 채운다. 정작 자주 쓰던 유용한 페이지들이 밀려나는 것이다. 이게 내가 겪은 문제였다.

## InnoDB의 개선된 LRU

InnoDB는 이 문제를 해결하기 위해 **Midpoint Insertion Strategy**를 사용한다.

새로 읽은 페이지를 LRU 리스트의 **중간 지점**에 삽입한다. 리스트가 Old 영역과 New 영역으로 나뉘는데:

- **Old 영역**: 새로 읽은 페이지가 처음 들어오는 곳. 1초 후에 다시 접근하면 New 영역으로 이동
- **New 영역**: 자주 사용되는 핵심 페이지들. 실제 LRU가 동작하는 곳

단순 테이블 스캔으로 읽힌 페이지는 Old 영역에 들어왔다가 1초 안에 재접근이 없으면 그냥 퇴출된다. 자주 쓰던 New 영역 데이터는 안전하게 유지된다.

## 알면 유용한 세 가지 최적화 기법

### 1. Adaptive Hash Index

특정 쿼리가 4번 이상 반복되면 MySQL이 자동으로 해시 인덱스를 만들어준다.

```
일반 B+ 트리 탐색: O(log n)
Adaptive Hash Index: O(1)
```

자주 실행되는 포인트 쿼리는 이 덕분에 엄청 빨라진다. 단, 메모리를 더 씁니다. 항상 트레이드오프.

### 2. Change Buffer

Secondary Index에 대한 삽입/삭제/수정 작업을 지연 처리하는 기능이다.

```
변경하려는 페이지가 버퍼풀에 없을 때
→ 디스크에 바로 쓰지 않고
→ Change Buffer에 잠시 저장
→ 나중에 배치로 한 번에 처리
```

랜덤 I/O를 순차 I/O로 바꿔주는 효과. 대량 INSERT 배치 작업에서 성능이 크게 개선된 이유가 이것 때문이었다.

### 3. Linear Read-Ahead (예측 읽기)

순차 읽기 패턴을 감지하면 다음 페이지들을 미리 로드한다. 대용량 테이블 스캔에서 효과적이다.

## 버퍼풀 히트율 모니터링

실제 운영 환경에서는 이 쿼리로 버퍼풀이 제대로 활용되고 있는지 확인한다.

```sql
SELECT
  FORMAT(pool_size * 16 / 1024, 0) as pool_size_MB,
  FORMAT(database_pages * 16 / 1024, 0) as data_size_MB,
  FORMAT((pool_size - database_pages) * 16 / 1024, 0) as free_MB,
  CASE
    WHEN (1 - (disk_reads / buffer_pool_reads)) * 100 >= 95
      THEN CONCAT(FORMAT((1 - (disk_reads / buffer_pool_reads)) * 100, 2), '% 🟢 매우 좋음')
    WHEN (1 - (disk_reads / buffer_pool_reads)) * 100 >= 90
      THEN CONCAT(FORMAT((1 - (disk_reads / buffer_pool_reads)) * 100, 2), '% 🟡 양호')
    ELSE
      CONCAT(FORMAT((1 - (disk_reads / buffer_pool_reads)) * 100, 2), '% 🔴 튜닝 필요')
  END as hit_rate
FROM information_schema.INNODB_BUFFER_POOL_STATS;
```

**90% 이상**을 유지하는 게 목표. 90% 미만이면 버퍼풀 크기를 늘리거나 자주 쓰지 않는 데이터를 정리해야 한다.

## 실제로 겪은 문제 해결

ETL 배치가 실행될 때마다 서비스 응답 속도가 느려졌던 문제. 원인은 배치의 풀 스캔이 버퍼풀을 오염시키는 것이었다.

해결 방법:
1. 배치 쿼리에 인덱스가 적용되도록 리팩토링
2. 버퍼풀 크기 증가 (innodb_buffer_pool_size 조정)
3. 배치 실행 시간을 트래픽이 적은 새벽으로 이동

버퍼풀의 동작 원리를 알고 나서야 왜 이런 현상이 생기는지 이해할 수 있었다. 항상 내부를 알고 쓰는 것과 모르고 쓰는 것은 다르다.
