배포 파이프라인에서 SSL 인증서 갱신 작업을 맡게 됐다.

`certbot renew` 명령어 실행하면 끝 아닌가 했는데, 왜 인증서가 필요한지, 왜 연간 갱신을 해야 하는지 도통 이해가 안 됐다. 그냥 외우는 것보다 원리를 알고 싶었다.

## 암호화의 두 가지 방법

### 대칭키 (Symmetric Key)

잠그는 키와 여는 키가 같다.

```
평문 "Hello" → [암호화: Key A] → 암호문 "Xk3#"
암호문 "Xk3#" → [복호화: Key A(동일)] → 평문 "Hello"
```

**장점:** 연산이 빠르다. AES-128 같은 대칭키는 성능이 좋다.

**단점:** 키를 어떻게 상대방에게 전달하는가? 인터넷으로 키를 보내면 도청당할 수 있다. 키 자체가 탈취되면 모든 암호화가 무의미하다.

### 비대칭키 (Asymmetric Key)

잠그는 키(Public Key)와 여는 키(Private Key)가 다르다. 한 쌍으로 작동한다.

```
Public Key로 암호화  →  Private Key로만 복호화 가능
Private Key로 암호화 →  Public Key로만 복호화 가능
```

**동작 원리 (수학적으로):**
```
평문 65 (ASCII 'A')
Public Key: (5, 323)  → 65^5 mod 323 = 12  (암호문)
Private Key: (29, 323) → 12^29 mod 323 = 65  (복호화)
```

소수를 기반으로 한 나머지 연산. 역산이 어렵다는 수학적 성질 덕분에 안전하다. 실제는 키 길이가 2048비트 이상이라서 브루트포스가 불가능하다.

**단점:** 연산이 느리다. RSA-2048은 AES-128보다 훨씬 느리다.

## HTTPS가 두 가지를 같이 쓰는 이유

**비대칭키만 쓰면:** 느려서 실용적이지 않다.
**대칭키만 쓰면:** 키 전달 문제.

HTTPS는 두 가지를 조합한다.

```
1. 비대칭키로 안전하게 대칭키를 교환한다
2. 이후 통신은 교환된 대칭키로 한다
```

## SSL 핸드셰이크 과정

```
Client                              Server
  |                                    |
  |──── Client Hello ─────────────────>|  (지원하는 암호화 목록 전송)
  |<─── Server Hello + 인증서 ─────────|  (서버 Public Key 포함된 인증서)
  |                                    |
  | [인증서 검증 — CA가 서명한 게 맞나?]  |
  |                                    |
  |──── Pre-Master Secret ────────────>|  (서버 Public Key로 암호화해서 전송)
  |                                    |
  | [양쪽에서 Pre-Master Secret으로    |
  |  동일한 세션 키 생성]               |
  |                                    |
  |<══ 이제부터 세션 키(대칭키)로 통신 ══>|
```

핵심은 **Pre-Master Secret을 서버 Public Key로 암호화해서 전달**하는 것이다. 서버만 Private Key를 갖고 있으니 서버만 복호화할 수 있다. 이렇게 공유된 대칭키로 이후 통신을 한다.

## 인증서가 필요한 이유 — 중간자 공격

비대칭키만으로는 한 가지 문제가 있다.

```
내가 네이버에 접속한다
  ↓
중간에 공격자가 끼어든다
  ↓
공격자가 자기 Public Key를 "네이버 Public Key"라고 속인다
  ↓
내가 공격자 Public Key로 암호화한다
  ↓
공격자가 복호화해서 내용을 읽는다
```

어떻게 "이 Public Key가 진짜 네이버 것임"을 보장하나?

**CA(Certificate Authority)가 서명한 인증서를 사용한다.**

```
네이버가 CA(예: DigiCert)에 신청:
  "이 Public Key가 naver.com 것입니다"

CA가 검증 후:
  [네이버 Public Key + 도메인 정보]를 CA Private Key로 서명

→ 이게 SSL 인증서
```

브라우저는 신뢰하는 CA 목록을 내장하고 있다. 인증서의 서명을 CA Public Key로 검증해서 "이 인증서가 진짜 CA가 발급한 것"인지 확인한다.

## 인증서 갱신이 필요한 이유

인증서에 유효기간이 있다. 보통 1년 또는 3개월(Let's Encrypt).

**왜 유효기간이 있나:**
- 개인키가 유출될 경우 피해 기간 제한
- CA가 발급 정책이나 알고리즘을 개선할 때 강제 갱신
- 실제 도메인 소유 여부를 주기적으로 재확인

CI/CD 파이프라인에 인증서 만료일 모니터링과 자동 갱신 스크립트를 추가한 이유가 여기 있다.

```bash
# certbot으로 자동 갱신 (cron으로 설정)
0 0 * * * /usr/bin/certbot renew --quiet --post-hook "systemctl reload nginx"
```

## DevOps에서 실용적인 포인트

**로드밸런서에서 TLS 종료(Termination):**

```
[Client] ── HTTPS ──> [Load Balancer] ── HTTP ──> [앱 서버들]
```

Load Balancer에서 SSL 처리를 담당하고, 내부 통신은 HTTP로. 앱 서버는 인증서 관리 불필요.

**mTLS (상호 TLS):**

서버 인증뿐 아니라 클라이언트도 인증서로 인증. 서비스 간 통신 보안에 사용. Kubernetes의 서비스 메시(Istio 등)가 주로 활용.

SSL/TLS 구조를 이해하고 나니 설정 파일의 옵션들이 무슨 의미인지 드디어 이해가 됐다.
