ETL 시스템에서 특정 테이블이 수억 건을 넘어가기 시작하더니 쿼리가 느려지기 시작했다.

인덱스도 걸려있는데 왜 느려? 라고 생각했는데, 인덱스 자체의 크기가 너무 커져서 메모리에 올라오지 못하는 상황이었다. 이때부터 수평적 확장에 대해 진지하게 공부했다.

## 레플리케이션 — 읽기 부하를 분산하자

**레플리케이션**은 하나의 Primary DB를 여러 Replica(Secondary)로 복제하는 것이다.

```
Primary DB  →  Replica 1
            →  Replica 2
            →  Replica 3
```

Primary는 쓰기 담당, Replica들은 읽기 담당으로 분리한다.

**장점:**
- 읽기 트래픽을 여러 DB로 분산
- Primary 장애 시 Replica를 Primary로 승격 가능 (고가용성)

**단점:**
- 복제 지연(Replication Lag) 발생 가능
- 쓰기는 여전히 Primary 하나에 집중됨

레플리케이션만으로는 쓰기 성능을 올릴 수 없다. 데이터 자체가 너무 많아지는 문제도 해결 안 된다.

## 파티셔닝 — 한 DB 안에서 테이블을 쪼개자

**파티셔닝**은 하나의 DB 안에서 테이블을 물리적으로 분할하는 것이다.

```sql
-- 날짜 기준 파티셔닝 예시
CREATE TABLE orders (
  id BIGINT,
  created_at DATE,
  ...
) PARTITION BY RANGE (YEAR(created_at)) (
  PARTITION p2023 VALUES LESS THAN (2024),
  PARTITION p2024 VALUES LESS THAN (2025),
  PARTITION p2025 VALUES LESS THAN (2026)
);
```

2023년 데이터를 조회하면 `p2023` 파티션만 스캔한다. 나머지 파티션은 **Partition Pruning**으로 제외된다.

**파티셔닝 종류:**
- **Range**: 범위 기준 (날짜, 숫자 범위)
- **List**: 특정 값 목록 기준 (지역 코드, 카테고리)
- **Hash**: 해시 함수 기준 (균등 분산)
- **Key**: MySQL이 선택한 해시 함수 기준

여러 파티션에 걸친 쿼리는 성능이 떨어진다. 파티션 키를 WHERE 조건에 꼭 포함해야 한다.

## 샤딩 — 여러 DB 서버로 쪼개자

파티셔닝이 한 서버 안에서 테이블을 나누는 것이라면,
**샤딩**은 아예 다른 DB 서버들로 데이터를 분산시키는 것이다.

```
User ID 1~100만  →  Shard 1 (DB 서버 1)
User ID 101~200만 →  Shard 2 (DB 서버 2)
User ID 201~300만 →  Shard 3 (DB 서버 3)
```

각 샤드는 독립적인 DB 서버. 쓰기 성능도 수평으로 확장 가능해진다.

**단점이 상당하다:**
- 크로스 샤드 조인이 매우 어렵거나 불가능
- 트랜잭션이 여러 샤드에 걸치면 분산 트랜잭션 필요
- 샤드 간 데이터 리밸런싱이 복잡

토스 같은 금융 서비스에서 5천억 건을 처리하는 데 샤딩이 핵심이었을 텐데, 이걸 직접 구현하는 비용이 엄청나다는 걸 알게 됐다.

## 데이터 압축과 아카이빙

데이터가 쌓이면 저장 비용도 문제다.

**MySQL InnoDB 압축:**
```sql
CREATE TABLE archived_logs (
  ...
) ROW_FORMAT=COMPRESSED KEY_BLOCK_SIZE=8;
```

압축률은 보통 50~70%지만, 압축/해제 CPU 비용이 발생한다.

**아카이빙 전략:**
- 일정 기간 지난 데이터를 아카이브 테이블로 이동
- 서비스 테이블은 최근 데이터만 유지 → 쿼리 성능 유지
- 아카이브 데이터는 콜드 스토리지로 (S3 등)

## 실제로 적용한 것

우리 시스템에서는 결국:
1. **레플리케이션**: 읽기 트래픽 분산
2. **날짜 기준 파티셔닝**: 이력 데이터 쿼리 성능 개선
3. **오래된 데이터 아카이빙**: 서비스 테이블 경량화

샤딩은 아직 그 규모는 아니어서 도입하지 않았다. 오버 엔지니어링도 문제니까.

"지금 필요한 만큼만" 이라는 원칙이 중요하다는 걸 다시 한번 느꼈다.
