좋아요 기능 설계를 처음 주도했을 때, 나는 단순히 `UPDATE article_like_count SET count = count + 1 WHERE article_id = ?` 이렇게 처리하면 되는 줄 알았다.

그리고 잘 돌아갔다. 혼자 테스트할 때는.

## 동시 요청에서 무슨 일이 일어나는가

두 요청이 동시에 들어온다고 가정하자.

```
좋아요 수 현재값: 0

트랜잭션 1: count 조회 → 0
트랜잭션 2: count 조회 → 0 (트랜잭션1이 아직 커밋 안 함)

트랜잭션 1: count = 0 + 1 = 1로 업데이트, 커밋
트랜잭션 2: count = 0 + 1 = 1로 업데이트, 커밋

결과: 좋아요 수 = 1 (실제로는 2가 되어야 함)
```

두 개의 트랜잭션이 정상적으로 실행됐지만 좋아요 수 증가 하나가 유실됐다. 트랜잭션이 있어도 동시성 문제는 해결되지 않는다.

## 해결책 세 가지

### 1. 비관적 락 (Pessimistic Lock)

"항상 충돌이 발생할 것이다"고 가정하고 미리 락을 잡는다.

**방법 1: 직접 UPDATE 쿼리**

```sql
UPDATE article_like_count
SET count = count + 1
WHERE article_id = :articleId
```

현재 DB에 저장된 값 기준으로 직접 연산한다. 이 UPDATE 구문이 실행되는 순간 레코드 락이 잡히고, 다른 트랜잭션은 이 락이 해제될 때까지 대기한다.

장점: 락을 잡는 시간이 짧다.

**방법 2: SELECT FOR UPDATE**

```java
// JPA에서 비관적 락으로 조회
@Lock(LockModeType.PESSIMISTIC_WRITE)
Optional<ArticleLikeCount> findByArticleId(Long articleId);

// 조회된 시점부터 락 점유
ArticleLikeCount likeCount = articleLikeCountRepository
    .findByArticleId(articleId)
    .orElseThrow();

// 객체 지향스럽게 증감 처리
likeCount.increase();
```

조회 시점부터 락을 잡기 때문에 JPA 엔티티를 객체로 다룰 수 있다. 대신 락 점유 시간이 더 길다.

### 2. 낙관적 락 (Optimistic Lock)

"충돌이 없을 것이다"고 가정하고 버전으로 충돌을 감지한다.

```java
@Version
private Long version;
```

버전 컬럼을 두고 업데이트 조건에 버전을 포함시킨다:

```sql
UPDATE article_like_count
SET count = count + 1, version = version + 1
WHERE article_id = :articleId AND version = :expectedVersion
```

트랜잭션 1이 version=1에서 version=2로 이미 업데이트한 상태라면, 트랜잭션 2의 업데이트 조건 `version = 1`이 일치하지 않아 실패한다. 이게 충돌 감지다.

JPA에서는 `ObjectOptimisticLockingFailureException`이 발생한다. 이걸 잡아서 재시도하거나 실패 응답을 내린다.

### 3. 비동기 순차 처리

요청을 큐에 넣고 게시글 단위로 순차 처리. 동시성 문제 자체가 사라진다. 하지만:
- 즉각 응답이 아니다 (비동기)
- 큐 시스템 구축 비용이 든다
- 실패 시 보상/재시도 로직이 필요하다

## 실제 선택

게시글 단위 좋아요 쓰기 트래픽은 파티션이 나뉘어 있어서 단일 레코드 경쟁이 생각보다 크지 않다.

비동기 순차 처리는 오버 엔지니어링이 될 가능성이 높다.

결국 가장 단순한 **비관적 락 방법 1** (직접 UPDATE 쿼리)을 선택했다. 짧은 락 점유 + 구현 단순함.

낙관적 락은 충돌이 드물고 재시도 로직이 간단한 케이스에 더 적합하다. 좋아요처럼 동일 게시글에 대한 동시 쓰기가 많을 수 있으면 충돌이 잦아서 재시도가 계속 발생할 수 있다.

## 회고

"트랜잭션 쓰면 다 해결된다"는 생각이 얼마나 위험한지 이 기능 하나로 뼈저리게 느꼈다.

트랜잭션은 원자성과 격리성을 보장하지만, 격리 수준과 락 전략에 따라 동시성 문제는 여전히 발생한다. 락의 종류와 동작 원리를 이해하지 않으면 정합성을 보장할 수 없다.
