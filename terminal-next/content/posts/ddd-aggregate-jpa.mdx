`@Entity`를 붙인 클래스가 도메인 모델이라고 생각했다.

틀렸다.

JPA 엔티티는 영속성 기술을 위한 클래스고, 도메인 모델은 비즈니스 개념을 표현하는 것이다. 이 둘이 합쳐질 수는 있지만, 같은 개념은 아니다.

## 애그리거트 (Aggregate)란

DDD에서 **애그리거트**는 관련된 엔티티와 값 객체를 묶어서 **하나의 일관성 단위**로 다루는 것이다.

주문(Order) 애그리거트를 예로 들면:

```
Order (Aggregate Root)
├── OrderId (Value Object)
├── CustomerId (Value Object)
├── OrderStatus (Value Object)
└── List<OrderItem> (Entity)
     ├── ProductId (Value Object)
     ├── Quantity (Value Object)
     └── Price (Value Object)
```

**Aggregate Root**인 `Order`를 통해서만 내부에 접근할 수 있다. `OrderItem`을 외부에서 직접 수정하는 건 안 된다.

## 애그리거트의 핵심: 불변식(Invariant) 보호

애그리거트가 하는 제일 중요한 일은 **비즈니스 규칙(불변식)을 항상 지키는 것**이다.

```java
public class Order {
    private List<OrderItem> items;
    private OrderStatus status;
    
    // 주문에 상품 추가 — 규칙을 Order 내부에서 강제
    public void addItem(Product product, int quantity) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("확정된 주문에는 상품을 추가할 수 없습니다");
        }
        if (quantity <= 0) {
            throw new IllegalArgumentException("수량은 1 이상이어야 합니다");
        }
        items.add(new OrderItem(product.getId(), quantity, product.getPrice()));
    }
    
    // 주문 확정 — 상태 전이 규칙
    public void confirm() {
        if (items.isEmpty()) {
            throw new IllegalStateException("주문 상품이 없습니다");
        }
        this.status = OrderStatus.CONFIRMED;
    }
}
```

비즈니스 규칙이 `Order` 클래스 안에 있다. 외부에서 `items.add()`를 직접 호출할 수 없다.

만약 JPA 엔티티에서 그냥 `getItems().add(item)`을 허용하면? 규칙을 우회할 수 있다. 일관성이 깨진다.

## JPA와 함께 쓰기

JPA를 쓰면서 도메인 모델을 유지하는 데 두 가지 접근이 있다.

### 방법 1: 도메인 모델 = JPA 엔티티

```java
@Entity
@Table(name = "orders")
public class Order {
    @Id
    private Long id;
    
    @Enumerated(EnumType.STRING)
    private OrderStatus status;
    
    @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderItem> items = new ArrayList<>();
    
    // 비즈니스 메서드도 같이
    public void addItem(Product product, int quantity) { ... }
}
```

장점: 간단하다.
단점: JPA 제약(기본 생성자 필요, Lazy Loading 등)이 도메인 모델 설계를 방해한다.

### 방법 2: 도메인 모델과 JPA 엔티티 분리

```java
// 순수 도메인 모델
public class Order {
    private final OrderId id;
    private final List<OrderItem> items;
    private OrderStatus status;
    
    // JPA 어노테이션 없음. 순수 자바.
}

// JPA 엔티티 (인프라 레이어)
@Entity
@Table(name = "orders")
public class OrderEntity {
    @Id
    private Long id;
    
    // 도메인 ↔ 엔티티 변환 메서드
    public static OrderEntity from(Order order) { ... }
    public Order toDomain() { ... }
}
```

장점: 도메인 모델이 JPA에 오염되지 않는다. 테스트가 순수하다.
단점: 변환 코드가 많아진다.

## 어떤 걸 선택할까

복잡도에 따라 다르다.

단순한 도메인이면 1번이 충분하다. JPA 어노테이션과 비즈니스 로직이 한 클래스에 있어도 관리 가능하다.

비즈니스 규칙이 복잡하고 도메인 로직 테스트가 중요하다면 2번을 고려한다.

실제 프로젝트에서는 핵심 애그리거트는 분리하고, 단순한 것들은 JPA 엔티티에 비즈니스 로직을 함께 두는 식으로 절충했다.

## 잘못 설계했을 때의 징조

- 비즈니스 로직이 Service 레이어에 산재해 있다
- 같은 규칙이 여러 Service에서 중복으로 구현된다
- 엔티티의 setter를 아무데서나 호출해서 상태를 바꾼다
- `getItems().add(item)` 같은 내부 컬렉션 직접 조작이 허용된다

이런 징조가 보이면 애그리거트 경계를 다시 생각해볼 때다.
