회사에서 서버 인프라 구성 업무가 생겼다. EC2, 로드밸런서, NAT Gateway가 나왔다.

AWS 콘솔을 보면서 "NAT Gateway가 왜 필요한 거지?"가 내내 궁금했다. 설정을 따라하면 되는데 왜 이런 구조인지 이해가 없으니까 문제가 생기면 손쓸 수가 없었다.

## 사설 IP와 공인 IP

우리 집 공유기를 생각하면 된다.

```
인터넷 (공인 IP: 123.456.789.0)
    ↓
공유기 (WAN: 123.456.789.0, LAN: 192.168.1.1)
    ↓
내 PC (192.168.1.100) / 폰 (192.168.1.101)
```

공인 IP는 인터넷에서 유일하다. 전 세계에 하나뿐.

사설 IP(`192.168.x.x`, `10.x.x.x`)는 내부 네트워크에서만 사용된다. 같은 사설 IP가 전 세계 수백만 개의 네트워크에서 사용된다.

**문제**: 사설 IP를 가진 기기가 어떻게 인터넷과 통신하는가?

## NAT: Network Address Translation

NAT가 사설 IP ↔ 공인 IP를 변환한다.

```
내 PC(192.168.1.100) → 요청 → 공유기/NAT → 인터넷 서버
내 PC(192.168.1.100) ← 응답 ← 공유기/NAT ← (공인 IP로 받아서 변환)
```

공유기(NAT 장비)가 테이블을 유지한다:

```
사설IP:포트          공인IP:포트       외부서버
192.168.1.100:50001  123.x.x.x:50001  → google.com:443
192.168.1.101:50002  123.x.x.x:50002  → naver.com:443
```

내 PC에서 구글로 요청이 나갈 때:
1. 공유기가 출발지를 공인 IP로 변경
2. 테이블에 매핑 기록
3. 구글에서 응답이 올 때 테이블 보고 사설 IP로 전달

여러 사설 IP가 하나의 공인 IP를 공유한다 (포트로 구분).

## AWS에서 NAT Gateway

```
퍼블릭 서브넷
├── 로드밸런서 (공인 IP 있음, 외부 접속 허용)
└── NAT Gateway (사설 → 공인 IP 변환)

프라이빗 서브넷
├── 앱 서버 (사설 IP만, 직접 외부 접속 불가)
└── DB 서버 (사설 IP만)
```

앱 서버는 프라이빗 서브넷에 있어서 외부에서 직접 접속할 수 없다. 외부 공격 노출 최소화.

근데 앱 서버가 외부 API를 호출해야 한다면? → NAT Gateway를 통해 나간다. 나갈 수는 있지만 들어올 수는 없다.

## HTTP vs HTTPS

HTTP 요청이 어떻게 전달되는지 네트워크 레벨에서 보면:

```
1. DNS 조회: naver.com → 223.130.195.200
2. TCP 3-way handshake (연결 수립)
3. HTTP 요청 전송 (평문)
4. HTTP 응답 수신 (평문)
5. TCP FIN (연결 종료)
```

HTTPS:
```
1. DNS 조회: naver.com → 223.130.195.200
2. TCP 3-way handshake
3. TLS 핸드셰이크 (인증서 검증, 대칭키 교환)
4. HTTPS 요청 전송 (암호화)
5. HTTPS 응답 수신 (암호화)
6. TCP FIN
```

HTTP는 중간에서 패킷을 캡처하면 내용이 보인다. HTTPS는 암호화되어 있어 키 없이는 읽을 수 없다.

## 로드밸런서 (Load Balancer)

```
클라이언트
    ↓
로드밸런서 (공인 IP 하나)
    ↓         ↓         ↓
서버 A      서버 B      서버 C
(사설 IP)  (사설 IP)  (사설 IP)
```

공인 IP 하나로 여러 서버에 분산한다. 서버가 추가되거나 하나가 죽어도 클라이언트는 로드밸런서 IP 하나만 알면 된다.

## DNS

`naver.com` → `223.130.195.200` 변환 과정:

```
1. 브라우저 캐시 확인
2. OS 캐시 확인 (/etc/hosts)
3. 로컬 DNS 서버 (ISP, 회사 DNS)
4. Root DNS 서버 → "com 담당 DNS 서버 주소 알려줘"
5. TLD DNS 서버 (.com) → "naver.com 담당 DNS 서버 주소 알려줘"
6. Authoritative DNS → "naver.com = 223.130.195.200"
```

각 단계에서 TTL(Time To Live) 동안 캐싱된다.

## 회고

"NAT Gateway가 왜 필요한데요?" → 이제 설명할 수 있다. 프라이빗 서브넷 서버들이 외부 API를 호출하기 위해 필요하다. 공인 IP 없이 나갈 수 없으니까.

"로드밸런서가 SSL을 처리한다는 게 무슨 말이에요?" → TLS Termination이다. 로드밸런서에서 HTTPS를 받아서 복호화하고, 내부 서버로는 HTTP로 전달한다. 서버들이 SSL 처리 안 해도 된다.

네트워크를 이해하면 인프라 구성도, 장애 디버깅도 달라진다.
