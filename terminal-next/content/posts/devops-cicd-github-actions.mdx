수동 배포의 끝은 항상 같다. 언젠가는 실수한다.

`git pull` → `./gradlew build` → `jar 파일 복사` → `서버 재시작`. 이 과정을 반복하다가 어느 날 개발 브랜치를 운영 서버에 배포해버렸다. 30분 동안 운영 장애.

그 날 이후로 CI/CD를 제대로 셋업하기로 했다.

## CI/CD가 해결하는 것

**CI (Continuous Integration):** 코드를 push할 때마다 자동으로 빌드 + 테스트. 문제를 빨리 발견한다.

**CD (Continuous Delivery/Deployment):** 검증된 코드를 자동으로 배포.

```
코드 Push → 빌드 → 테스트 → Docker 이미지 빌드 → 배포
(이 모든 과정이 자동으로)
```

사람이 개입할 여지가 줄어들수록 실수도 줄어든다.

## GitHub Actions 기본 구조

```yaml
# .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    branches: [ main ]  # main 브랜치에 push될 때 실행

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up JDK 17
        uses: actions/setup-java@v3
        with:
          java-version: '17'
          distribution: 'temurin'
      
      - name: Run Tests
        run: ./gradlew test
  
  build-and-push:
    needs: test  # test job이 성공해야 실행
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Build Docker Image
        run: |
          docker build -t myapp:${{ github.sha }} .
          docker tag myapp:${{ github.sha }} myregistry/myapp:latest
      
      - name: Push to Registry
        run: |
          docker login -u ${{ secrets.DOCKER_USERNAME }} \
                       -p ${{ secrets.DOCKER_PASSWORD }}
          docker push myregistry/myapp:latest
  
  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Server
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            docker pull myregistry/myapp:latest
            docker stop myapp || true
            docker run -d --name myapp -p 8080:8080 myregistry/myapp:latest
```

## Docker로 환경 일관성 보장

로컬에서 되는데 서버에서 안 된다 — Java 버전이 다르거나, 환경변수가 다르거나.

Docker는 이 문제를 해결한다. 빌드 환경을 이미지로 패키징해서 어디서든 동일하게 실행.

```dockerfile
# Dockerfile
FROM eclipse-temurin:17-jre-alpine

WORKDIR /app

COPY build/libs/*.jar app.jar

# 컨테이너 실행 시 환경변수를 외부에서 주입
ENV SPRING_PROFILES_ACTIVE=prod

EXPOSE 8080

ENTRYPOINT ["java", "-jar", "app.jar"]
```

**멀티 스테이지 빌드** — 빌드 환경과 실행 환경 분리:

```dockerfile
# 빌드 스테이지 (JDK 필요)
FROM eclipse-temurin:17-jdk-alpine AS build
WORKDIR /app
COPY . .
RUN ./gradlew build -x test

# 실행 스테이지 (JRE만 있어도 됨 → 이미지 크기 줄어듦)
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/build/libs/*.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
```

## 무중단 배포

`docker stop` → `docker run`을 하면 잠깐 서비스가 내려간다. Blue-Green 배포로 해결.

```bash
# 현재 Blue(포트 8080)가 운영 중일 때
# Green(포트 8081)에 새 버전 배포

docker run -d --name myapp-green -p 8081:8080 myregistry/myapp:latest

# Green 헬스체크
until curl -f http://localhost:8081/actuator/health; do
  sleep 2
done

# Nginx 트래픽을 Green으로 전환
nginx -s reload  # upstream을 8081로 변경 후

# 이전 Blue 컨테이너 제거
docker rm -f myapp-blue

# Green을 Blue라고 리네임
docker rename myapp-green myapp-blue
```

요즘은 Nginx 대신 Kubernetes의 Rolling Update나 Argo Rollouts를 더 많이 씀.

## 시크릿 관리

GitHub Secrets에 민감한 정보들을 저장한다.

```yaml
# 절대 코드에 직접 쓰면 안 됨
env:
  DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
  JWT_SECRET: ${{ secrets.JWT_SECRET }}
```

Spring Boot에서 환경변수로 주입:

```yaml
# application-prod.yml
spring:
  datasource:
    password: ${DB_PASSWORD}  # 환경변수에서 주입
```

## 모니터링과 알림

배포가 실패하면 빠르게 알아야 한다.

```yaml
# GitHub Actions에서 Slack 알림
- name: Notify Slack on Failure
  if: failure()
  uses: 8398a7/action-slack@v3
  with:
    status: failure
    webhook_url: ${{ secrets.SLACK_WEBHOOK }}
    text: "🚨 배포 실패! ${{ github.repository }}의 ${{ github.sha }}"
```

배포 후 서버 헬스체크:

```yaml
- name: Health Check
  run: |
    sleep 10  # 서버 시작 대기
    response=$(curl -s -o /dev/null -w "%{http_code}" http://myserver/actuator/health)
    if [ "$response" != "200" ]; then
      echo "Health check failed: $response"
      exit 1
    fi
```

## 회고

CI/CD를 셋업하면서 가장 크게 느낀 것: **신뢰할 수 있는 배포 프로세스**가 생겼다는 것.

이전에는 배포할 때마다 혹시 실수하지 않았나 불안했다. 이제는 `git push`하면 나머지는 자동으로 처리되고, 실패하면 알림이 온다.

초기 셋업에 시간이 걸리지만, 장기적으로 얻는 이득이 훨씬 크다. 특히:
- 배포 실수 → 거의 0에 수렴
- 배포 시간 → 평균 15분 → 3분으로
- 배포 두려움 → 자주 배포하게 됨 (소규모 변경으로)

"자주 배포하면 할수록 각 배포가 작아지고, 작을수록 문제가 생겨도 더 쉽게 롤백할 수 있다." 이 말이 이제야 실감이 난다.
