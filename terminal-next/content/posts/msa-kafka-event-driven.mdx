주문 처리 로직이 점점 복잡해졌다.

```java
// 주문 완료 시 처리해야 할 것들
public void completeOrder(Long orderId) {
    orderService.complete(orderId);
    paymentService.charge(orderId);        // 결제
    emailService.sendConfirmEmail(orderId); // 이메일
    pointService.addPoints(orderId);        // 포인트
    inventoryService.decrease(orderId);     // 재고
}
```

각 서비스를 Feign Client로 동기 호출하고 있었다. 문제가 생기기 시작했다.

## 동기 호출의 문제

**문제 1: 장애 전파**

Email Service가 죽으면? `completeOrder()`가 이메일 발송 단계에서 예외를 던진다. 주문은 완료됐는데 결제까지 롤백해야 할까?

**문제 2: 성능**

이메일 발송이 느린 외부 SMTP를 쓴다면? 사용자가 주문 완료 응답을 받기까지 이메일 발송 시간만큼 기다려야 한다.

**문제 3: 강결합**

Order Service가 Email Service, Point Service, Inventory Service를 직접 알아야 한다. 새로운 처리가 추가될 때마다 Order Service를 수정해야 한다.

## 이벤트 기반으로 전환

Kafka를 도입했다. Order Service는 주문 완료 이벤트만 발행한다.

```java
// Order Service: 이벤트 발행
public void completeOrder(Long orderId) {
    Order order = orderRepository.findById(orderId);
    order.complete();
    orderRepository.save(order);
    
    // 이벤트 발행 — 누가 처리할지 모름. 신경 안 씀.
    kafkaTemplate.send("order-completed", new OrderCompletedEvent(orderId));
    
    // 즉시 응답 반환
}
```

각 서비스가 자기 관심사에 맞는 이벤트를 구독한다.

```java
// Email Service: 주문 완료 이벤트 구독
@KafkaListener(topics = "order-completed")
public void handleOrderCompleted(OrderCompletedEvent event) {
    emailService.sendConfirmEmail(event.getOrderId());
}

// Point Service: 동일 이벤트 구독
@KafkaListener(topics = "order-completed")
public void handleOrderCompleted(OrderCompletedEvent event) {
    pointService.addPoints(event.getOrderId());
}

// Inventory Service: 동일 이벤트 구독
@KafkaListener(topics = "order-completed")
public void handleOrderCompleted(OrderCompletedEvent event) {
    inventoryService.decrease(event.getOrderId());
}
```

Order Service는 더 이상 다른 서비스를 직접 의존하지 않는다.

## Kafka의 내부 구조

```
Producer → [Topic: order-completed] → Consumer Group A (Email Service)
                                    → Consumer Group B (Point Service)
                                    → Consumer Group C (Inventory Service)
```

**Topic**: 메시지 카테고리. `order-completed`라는 파이프.

**Partition**: Topic을 물리적으로 나눈 것. 병렬 처리 가능.

**Consumer Group**: 동일 Group ID를 가진 Consumer들. 파티션을 나눠서 처리.

```
Partition 0 → Consumer A-1
Partition 1 → Consumer A-2
Partition 2 → Consumer A-3
```

한 파티션은 같은 Consumer Group 내에서 하나의 Consumer만 처리한다. 메시지 순서가 보장된다.

## 데이터 동기화 문제

MSA에서 각 서비스가 자체 DB를 가진다. Order DB에 있는 정보를 User Service에서 보여줘야 할 때?

매번 Order Service에 API 호출하는 대신, Kafka로 데이터를 동기화한다.

```java
// Order Service: 주문 생성 시 이벤트 발행
kafkaTemplate.send("order-events", new OrderCreatedEvent(order));

// User Service: 본인 DB에 주문 정보 저장
@KafkaListener(topics = "order-events")
public void syncOrderData(OrderCreatedEvent event) {
    userOrderRepository.save(UserOrder.from(event));
}
```

실시간 정합성은 포기하지만 (Eventually Consistent), User Service가 Order Service에 의존하지 않아도 된다.

## 겪은 트러블들

**메시지 중복 처리:** 네트워크 문제로 같은 메시지가 두 번 처리되는 경우가 있었다. Consumer를 Idempotent하게 만들어야 한다.

```java
// 처리된 이벤트 ID를 저장해서 중복 방지
if (processedEventRepository.existsById(event.getEventId())) {
    return; // 이미 처리됨
}
processedEventRepository.save(new ProcessedEvent(event.getEventId()));
// 실제 처리...
```

**메시지 순서 보장:** 같은 주문의 이벤트가 순서대로 처리되어야 할 때는 같은 주문 ID를 Partition Key로 사용했다. 같은 키는 항상 같은 파티션으로 간다.

## 회고

Kafka 도입 후 Order Service 코드가 훨씬 단순해졌다. 하지만 전체 시스템 이해는 더 어려워졌다.

"주문 완료했는데 포인트가 왜 안 쌓이지?" 같은 문제를 디버깅할 때 Kafka 토픽, Consumer 상태, 오프셋 등을 전부 확인해야 한다.

복잡성이 사라지는 게 아니라 다른 곳으로 이동하는 것 같다. 그 복잡성을 어디에 두느냐의 선택이 아키텍처다.
