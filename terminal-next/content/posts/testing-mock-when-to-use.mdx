처음 테스트 코드를 작성할 때 무조건 `@MockBean`으로 외부 의존성을 다 모킹했다.

*빠르고 간단하잖아.*

근데 시간이 지나면서 문제가 생겼다. 테스트는 다 통과하는데 실제 서비스에서 버그가 나는 상황. Mock이 실제 동작과 달랐던 것이다.

## Mock이 필요한 이유

테스트 중에 외부 시스템에 실제로 접근하면 문제가 생긴다.

```java
@Test
void 이메일_발송_테스트() {
    // 테스트할 때마다 실제 이메일이 발송된다?!
    emailService.sendWelcomeEmail("test@example.com");
}
```

- 실제 DB에 데이터가 쌓인다
- 외부 API 호출 비용이 발생한다
- 네트워크 상태에 따라 테스트 결과가 달라진다

이런 상황에서 Mock은 외부 의존성을 가짜로 대체해서 빠르고 독립적인 테스트를 가능하게 한다.

## Mockito 기본 사용법

```java
// Mock 객체 생성
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private OrderRepository orderRepository; // 가짜 Repository
    
    @Mock
    private EmailService emailService; // 가짜 이메일 서비스
    
    @InjectMocks
    private OrderService orderService; // Mock을 주입받은 실제 서비스
    
    @Test
    void 주문_확정시_이메일이_발송된다() {
        // given
        Order order = createTestOrder();
        given(orderRepository.findById(1L)).willReturn(Optional.of(order));
        
        // when
        orderService.confirmOrder(1L);
        
        // then
        verify(emailService, times(1)).sendOrderConfirmEmail(any());
    }
}
```

## Mock을 써야 할 때

### 1. 외부 API, 이메일, SMS 서비스
```java
@Mock
private SlackNotifier slackNotifier; // Slack API 실제 호출 X
```

### 2. 시간에 의존하는 로직
```java
@Mock
private Clock clock;
given(clock.now()).willReturn(LocalDateTime.of(2025, 3, 15, 10, 0));
```

### 3. 테스트하기 어려운 랜덤/외부 요소
```java
@Mock
private RandomCodeGenerator codeGenerator;
given(codeGenerator.generate()).willReturn("FIXED-CODE");
```

## Mock을 쓰면 안 될 때 — 내가 실수한 패턴

### 실수 1: Repository를 항상 Mock으로 처리

```java
// 이렇게 하면 뭘 테스트하는 건가?
@Mock
private OrderRepository orderRepository;

@Test
void 주문_저장_테스트() {
    given(orderRepository.save(any())).willReturn(testOrder);
    
    Order saved = orderService.createOrder(command);
    
    // Repository를 Mock했으니 실제 저장은 안 됨
    // 그냥 내가 설정한 Mock을 반환할 뿐
    assertThat(saved).isEqualTo(testOrder); // 의미 없는 테스트
}
```

이런 테스트는 실제 DB 동작을 전혀 검증하지 않는다.

### 해결: 통합 테스트에서는 실제 DB 사용

```java
@DataJpaTest
class OrderRepositoryTest {
    @Autowired
    private OrderRepository orderRepository; // 실제 Repository
    
    @Test
    void 주문을_저장하고_조회한다() {
        Order order = createOrder();
        orderRepository.save(order);
        
        Optional<Order> found = orderRepository.findById(order.getId());
        assertThat(found).isPresent();
    }
}
```

### 실수 2: 비즈니스 로직을 Mock으로 우회

```java
// 이건 그냥 비즈니스 로직을 테스트 안 하는 것
@Mock
private OrderPriceCalculator priceCalculator;
given(priceCalculator.calculate(any())).willReturn(10000);
```

비즈니스 로직은 가능한 한 실제로 테스트해야 한다.

## 좋은 테스트 전략

```
단위 테스트 (빠름, Mock 활용)
  ↓
통합 테스트 (@SpringBootTest or @DataJpaTest, 실제 DB)
  ↓
E2E 테스트 (전체 시스템 흐름 검증)
```

- **단위 테스트**: 비즈니스 로직만. 의존성은 Mock.
- **통합 테스트**: Repository, DB 쿼리. 실제 H2나 Testcontainers.
- **E2E 테스트**: 실제 HTTP 요청부터 DB까지.

## 기준

Mock을 쓸지 말지의 기준:

> **"이 Mock이 실제 동작을 충분히 대신하는가?"**

외부 API, 이메일 등 → Mock 써도 됨  
비즈니스 로직, DB 쿼리 → 가능한 실제로 테스트

Mock을 많이 쓸수록 테스트가 빨라지지만, 신뢰도는 낮아진다. 반대로 실제 환경에 가까울수록 느려지지만 신뢰도는 높아진다. 이 트레이드오프를 의식하면서 균형을 맞추는 게 핵심이다.
