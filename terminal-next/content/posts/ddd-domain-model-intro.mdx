새 프로젝트 킥오프 미팅에서 "DDD 적용해봅시다"라는 말이 나왔다.

그때 내 머릿속: *Entity 만들고, Value Object 만들고, Repository 만들면 되는 거지?*

3개월 후 회고 때 "왜 복잡성이 줄지 않았지?" 라는 질문이 나왔다. 패턴은 다 썼는데 코드는 여전히 엉켜있었다. 토비 선생님의 강의를 들으면서 내가 뭘 놓쳤는지 알게 됐다.

## 도메인이 뭔가

소프트웨어는 사용자의 어떤 활동, 업무, 관심사를 도와주기 위해 존재한다. 그 **특정 주제 영역**이 도메인이다.

- 금융 도메인
- 이커머스 도메인  
- 제조 도메인

개발자로서 "나는 금융 도메인에서 일했다"고 말할 때의 그 도메인이다.

## 도메인 모델이란

도메인은 현실 세계다. 복잡하고 자주 변한다. 그걸 그대로 코드에 옮길 수 없다.

**추상화**를 해야 한다. 핵심적이고 소프트웨어에서 중요한 것들만 뽑아내서 단순화하는 것. 그 결과물이 **도메인 모델**이다.

온라인 서점을 예로 들면:

```
도메인 핵심 개념 (명사):
  - 책 (Book)
  - 고객 (Customer)
  - 주문 (Order)
  - 장바구니 (Cart)
  
도메인 관계:
  - 고객이 책을 주문한다
  - 주문 전 장바구니에 담는다
  
도메인 규칙:
  - 주문 총액 = 상품 수량 × 가격의 합계
  - 재고가 없으면 주문할 수 없다
```

이것들을 뽑아내고 정리하는 것 자체가 도메인 모델링이다. 다이어그램이나 문서가 도메인 모델이 아니라, **팀 모두의 머릿속에 있는 이해**가 도메인 모델이다.

## DDD는 패턴이 아니다

내가 틀렸던 부분이 여기다.

DDD는 Aggregate, Entity, Value Object 같은 패턴 모음이 아니다.

> **도메인 모델을 만들어서 개발의 중심에 두는 접근 방법.**

복잡한 도메인을 다룰 때, 그 복잡성을 도메인 모델로 잘 정리하고, 그게 설계와 코드까지 이어지게 하는 것이 DDD다.

패턴들은 그 과정을 돕는 도구일 뿐이다.

## 모델 주도 설계 (Model-Driven Design)

기존 방식:
```
현업 분석가 → 분석 문서 → 설계자 → UML 다이어그램 → 개발자 → 코드
```

이렇게 끊어지면 중간에 도메인 지식이 왜곡되거나 사라진다. 분석할 때 있던 개념이 코드에는 없는 상황이 생긴다.

DDD가 말하는 것:
```
도메인 모델 ↔ 설계 ↔ 코드 (지속적으로 연결)
```

코드를 짜다가 도메인 모델이 바뀌어야 하면 바꾼다. 그 변화가 코드에도 즉각 반영된다. 반대도 마찬가지.

## Ubiquitous Language (보편 언어)

이게 DDD에서 가장 실용적이었던 개념이다.

현업 전문가, 기획자, 개발자가 **같은 용어로 같은 것을 이야기해야** 한다.

실제 경험한 혼란:
- 기획서에서는 "주문" → 코드에서는 `Order`
- 기획서에서는 "취소" → 코드에서는 `cancel`, `delete`, `remove` 혼용
- 현업에서 "승인" → 개발팀에서 "confirm"과 "approve" 혼용

이런 용어 불일치가 쌓이면 의사소통 비용이 폭증한다. 보편 언어를 만들면:

```
한국어 → 영어 매핑 용어 사전
'주문'  → Order
'주문취소' → OrderCancellation (cancel 아님!)
'배송'  → Delivery
'결제승인' → PaymentApproval
```

이걸 문서, 회의, 코드에 일관되게 쓴다.
