새 프로젝트 아키텍처 회의에서 "MSA로 가자"는 결론이 났다.

그때 솔직한 내 생각: *왜? 더 복잡해지잖아.*

기존 모놀리식으로 잘만 돌아가고 있었다. 굳이 나눠서 복잡도만 올리는 거 아닌가 싶었다. Spring Cloud 강의를 본격적으로 파면서 그 "왜"를 이해하게 됐다.

## 모놀리식 아키텍처가 괜찮을 때

```
[Client] → [단일 애플리케이션 (UI + Business Logic + DB Logic)] → [단일 DB]
```

초기에는 이게 훨씬 단순하고 좋다. 배포 하나로 끝나고, 트랜잭션 관리도 쉽고, 서비스 간 통신 비용도 없다.

작은 팀, 초기 서비스라면 모놀리스가 정답에 가깝다.

## 모놀리스가 힘들어지는 시점

문제는 서비스가 커지면서 시작된다.

```java
// 하나의 프로젝트에 모든 것이...
com.myapp
├── user/         // 800개 클래스
├── order/        // 1200개 클래스  
├── payment/      // 600개 클래스
├── notification/ // 400개 클래스
└── ...           // 계속 추가 중
```

**문제 1: 배포가 공포가 된다**

결제 로직 버그 하나 수정했는데 → 전체 빌드 → 전체 테스트 → 전체 배포.

알림 메시지 오타 하나 고치는데 결제 서비스까지 배포한다. 배포가 두려워지니 자주 못 한다. 기능 반영이 느려진다.

**문제 2: 장애가 전파된다**

결제 모듈에서 메모리 누수 → 전체 프로세스 메모리 부족 → 유저 서비스도 죽음.

아무 관련 없는 알림 처리 버그 하나가 전체 서비스를 멈출 수 있다.

**문제 3: 기술 선택이 경직된다**

처음에 Java 11, MySQL로 시작했으면 계속 그것만 써야 한다. 채팅 기능은 Node.js가 훨씬 나은데 전체를 Java로 해야 한다.

## MSA가 해결하려는 것

MSA는 이 문제들을 **서비스 분리**로 해결한다.

```
[Client]
    ↓
[API Gateway]
    ↓
[User Service] [Order Service] [Payment Service] [Notification Service]
      ↓               ↓              ↓                   ↓
   [User DB]      [Order DB]    [Payment DB]       [Notif DB]
```

- **독립 배포**: Order Service 수정 → Order Service만 배포
- **장애 격리**: Notification 서비스 다운 → 결제/주문은 계속 동작
- **기술 다양성**: 각 서비스가 최적의 기술 스택 선택 가능

## 근데 MSA는 공짜가 아니다

여기서 중요한 걸 놓쳤었다. MSA는 모놀리스의 문제를 해결하지만, 대신 새로운 복잡성을 가져온다.

**새로 생기는 문제들:**

- 서비스 간 네트워크 통신 → 실패, 타임아웃, 지연
- 분산 트랜잭션 → 결제 성공, 주문 실패하면?
- 서비스 디스커버리 → 서비스가 어디 있는지 어떻게 찾나?
- 배포 복잡도 → 10개 서비스를 어떻게 관리하나?
- 모니터링 → 장애가 어디서 났는지 추적하기 어려움

**Martin Fowler의 경고:**

> "MSA는 프로세스 경계가 서비스 단위로 나뉜다. 분산 시스템의 모든 복잡성을 감수해야 한다."

## 언제 MSA를 선택해야 하나

모놀리스가 실제로 이 문제들로 고통받고 있을 때:

- 배포가 너무 느리고 위험하다
- 특정 컴포넌트만 스케일 아웃이 필요하다
- 팀이 커져서 같은 코드베이스 작업이 충돌된다
- 서비스 일부 장애가 전체를 죽인다

"시작부터 MSA"는 오버 엔지니어링일 가능성이 높다.

## 회고

"왜 MSA를 도입하는가"를 명확하게 정의하지 않고 아키텍처 패턴만 따라간 결과였다. 모놀리스의 어떤 구체적인 문제를 풀기 위해 MSA를 쓰는지, 그걸 먼저 팀이 합의했어야 했다.

아키텍처는 트레이드오프다. 모놀리스의 문제를 해결하기 위해 MSA의 복잡성을 감수하는 것이 가치 있는 때가 있고, 아닌 때가 있다.
