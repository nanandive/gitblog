`@Autowired`나 `@Service`를 아무 생각 없이 붙이면 스프링이 알아서 해준다.

근데 "알아서"가 어떻게 동작하는지 모르면 설정 오류가 났을 때 손발이 묶인다. 토비의 스프링6 강의를 보면서 처음부터 다시 이해했다.

## IoC: Inversion of Control (제어의 역전)

일반적인 코드에서는 내가 객체를 만든다.

```java
// 일반 코드 — 내가 제어
public class OrderService {
    private UserRepository userRepository = new UserRepositoryImpl();
    private OrderRepository orderRepository = new OrderRepositoryImpl();
    
    public void placeOrder(Long userId, Order order) {
        User user = userRepository.findById(userId);
        orderRepository.save(order);
    }
}
```

`OrderService`가 `UserRepository`를 직접 만들고 제어한다.

IoC는 이 제어를 외부에 넘기는 것이다. 객체를 누가 만들지, 어떤 구현체를 쓸지를 외부(스프링)가 결정한다.

```java
// IoC — 스프링이 제어
@Service
public class OrderService {
    private final UserRepository userRepository;
    private final OrderRepository orderRepository;
    
    // 스프링이 주입해줌
    @Autowired
    public OrderService(UserRepository userRepository, OrderRepository orderRepository) {
        this.userRepository = userRepository;
        this.orderRepository = orderRepository;
    }
}
```

`OrderService`는 `UserRepository`가 어떤 구현체인지 모른다. 스프링이 알아서 넣어준다.

## DI: Dependency Injection (의존성 주입)

IoC를 구현하는 방법 중 하나가 DI다.

의존 관계를 외부에서 주입해준다. 스프링 컨테이너가 이 역할을 한다.

세 가지 방법:

```java
// 1. 생성자 주입 (권장)
@Service
public class OrderService {
    private final UserRepository userRepository;
    
    public OrderService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}

// 2. 세터 주입
@Service
public class OrderService {
    private UserRepository userRepository;
    
    @Autowired
    public void setUserRepository(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}

// 3. 필드 주입 (권장하지 않음)
@Service
public class OrderService {
    @Autowired
    private UserRepository userRepository;
}
```

생성자 주입을 권장하는 이유:
- `final` 키워드 → 불변성, 실수 방지
- 테스트 시 Mock 주입이 명확
- 순환 의존성이 컴파일 시점에 발견됨

필드 주입의 문제점:
- `final` 못 씀 → 런타임에 변경 가능
- 테스트 시 `@SpringBootTest` 없이 Mock 주입 어려움
- 리플렉션으로 주입하므로 스프링 없이 쓰기 어려움

## 스프링 빈 (Spring Bean)

스프링이 관리하는 객체를 빈(Bean)이라고 한다.

```java
@Component  // 스프링 빈으로 등록
public class UserRepositoryImpl implements UserRepository { ... }

@Service    // @Component의 특수화
public class OrderService { ... }

@Repository // @Component의 특수화 (+ 예외 변환 기능)
public class OrderRepositoryImpl implements OrderRepository { ... }

@Controller // @Component의 특수화 (+ MVC 처리)
public class OrderController { ... }
```

스프링 컨테이너(ApplicationContext)가 시작될 때 이 빈들을 모두 만들어서 의존 관계를 연결한다.

## 수동 빈 등록

```java
@Configuration
public class AppConfig {
    
    @Bean
    public UserRepository userRepository() {
        return new UserRepositoryImpl(dataSource());
    }
    
    @Bean
    public DataSource dataSource() {
        HikariDataSource ds = new HikariDataSource();
        ds.setJdbcUrl(System.getenv("DB_URL"));
        return ds;
    }
}
```

수동 등록은 주로 외부 라이브러리나 조건에 따라 다른 구현체를 주입할 때 사용한다.

## 스프링 없이 이해하기

스프링 없이 OrderService를 만들면:

```java
// 직접 의존 관계 조립
DataSource dataSource = new HikariDataSource(...);
UserRepository userRepository = new UserRepositoryImpl(dataSource);
OrderRepository orderRepository = new OrderRepositoryImpl(dataSource);
OrderService orderService = new OrderService(userRepository, orderRepository);
```

스프링이 이 일을 자동화해준다. 어떤 빈이 어떤 의존성이 필요한지 분석해서 자동으로 연결한다.

`@SpringBootApplication`이 있으면 해당 패키지 하위를 스캔해서 `@Component`가 붙은 클래스들을 모두 빈으로 등록한다.

## 회고

"스프링이 알아서 해준다"에서 멈추지 않고 "어떻게 알아서 해주는지"를 이해하고 나니 설정 문제를 디버깅할 수 있게 됐다.

예를 들어 `NoUniqueBeanDefinitionException`은 같은 타입의 빈이 두 개 이상 있을 때 발생한다. 이제는 `@Primary`나 `@Qualifier`로 어떤 빈을 주입할지 명시하면 해결된다는 걸 안다.

`@Autowired`를 이해한다는 건 스프링이 런타임에 어떻게 의존 관계를 연결하는지를 이해한다는 것이다.
